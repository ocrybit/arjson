# Weavepack v0.6 — planning document

**Status:** Planning. Defines the scope for the v0.6 cycle.
**Precondition:** v0.5 complete (2026-05-09). All v0.5 benchmark gates pass.

## What v0.6 is

**Profile #6: weavepack-graph** — the primary deliverable. A labeled
property-graph format with native node/edge delta chains. No existing
graph format has per-element column encoding or protocol-level delta
semantics; this is the gap.

## Why weavepack-graph

The V0.5-PLANNING.md explicitly deferred weavepack-graph to v0.6+,
noting that RDF/property-graph delta semantics require deeper research
and that there was no strong adopter demand at the time. Two conditions
have since shifted:

1. **The row-delta pattern is now proven in two profiles** (tabular v0.4,
   log v0.5). The edge-delta model for graphs is structurally similar
   to row-level deltas — an edge is a typed row in a src×dst column pair.
   The machinery is understood; graph is an application of it.

2. **Knowledge graph and graph-database workloads are growing** in
   AI/ML pipelines (entity graphs, knowledge retrieval, RAG backends).
   The ability to ship compressed delta frames for graph changes is
   directly useful here.

weavepack-graph's unique value proposition against GraphML + gzip and
RDF/Turtle + gzip:

| | typed | schema | deltas | column-enc | self-desc |
|---|---|---|---|---|---|
| GraphML | ✓ | ~ | ✗ | ✗ | ✓ |
| RDF/Turtle | ~ | ~ | ✗ | ✗ | ✓ |
| Neo4j CSV | ✓ | ✗ | ✗ | ✗ | ~ |
| JSON-LD | ✓ | ~ | ✗ | ✗ | ✓ |
| **weavepack-graph** | **✓** | **✓** | **✓** | **✓** | **✓** |

The win is sharpest for two scenarios:

1. **Evolving knowledge graphs** — entity-relationship graphs updated
   with insertion/deletion events. GraphML re-encodes the entire graph
   per snapshot. weavepack-graph emits `node_insert`, `edge_insert`, and
   `edge_delete` delta frames containing only the changed elements.

2. **High-cardinality homogeneous graphs** — e.g., social graphs where
   all nodes have the same property schema (user_id, name, country,
   joined_date). RDF and GraphML repeat property names on every element.
   weavepack-graph encodes property names once per schema and
   bit-packs property values; repeated or monotone properties
   compress to near-zero via RLE and delta-pack.

## Profile model

weavepack-graph is a **labeled property graph** profile:

- **Nodes** carry a `nid` (uint64, monotone), an optional label (interned
  string), and named typed property columns.
- **Edges** carry an `eid` (uint64, monotone), `src` and `dst` (uint64
  node refs), an optional label (interned string), and named typed
  property columns.
- Data is organized into typed **blocks** — a node_block groups nodes
  of the same label; an edge_block groups edges of the same label.
- Both block types use **tabular-compatible column encoding** (same
  ctypes, same null-bitmap layout, same bit-pack rules).
- A **graph document** is a sequence of node_blocks and edge_blocks,
  optionally preceded by a schema sidecar.

RDF triples can be mapped to weavepack-graph:

| RDF | weavepack-graph |
|---|---|
| subject IRI | nid (URI interned in strmap) |
| predicate IRI | edge label (interned in strmap) |
| object IRI | dst nid (URI interned in strmap) |
| object literal | edge property column |
| named graph | edge block label |

## Profile #6 spec plan

All documents live under `weavepack/profiles/graph/`.

### 00-overview.md

Context, motivation, incumbent comparison (GraphML, RDF/Turtle, Neo4j CSV,
JSON-LD), property-graph model summary.

### 01-types.md

Column types (superset of tabular ctypes):

| ctype | Name | Bits | Notes |
|---|---|---|---|
| 0 | `bool` | 1 | (inherited from tabular) |
| 1 | `int8` | 8 | signed |
| 2 | `int16` | 16 | signed |
| 3 | `int32` | 32 | signed |
| 4 | `int64` | 64 | signed |
| 5 | `uint8` | 8 | |
| 6 | `uint16` | 16 | |
| 7 | `uint32` | 32 | |
| 8 | `uint64` | 64 | |
| 9 | `float32` | 32 | IEEE 754 binary32, little-endian |
| 10 | `float64` | 64 | IEEE 754 binary64, little-endian |
| 11 | `string` | variable | LEB128-byte-count + UTF-8 bytes |
| 12 | `bytes` | variable | LEB128-byte-count + raw octets |
| 13 | `date32` | 32 | days since Unix epoch (1970-01-01) |
| 14 | `timestamp64` | 64 | microseconds since Unix epoch (UTC) |
| 15 | `node_id` | 64 | uint64 with semantic type "node reference" |

`node_id` (ctype 15) encodes the same 8 bytes little-endian as
`uint64` (ctype 8) but signals to a reader that the column's values
are node identifiers referencing the node table. This enables graph
traversal APIs to distinguish property values from structural references
without schema information.

Nullability: same as tabular — a companion null bitmap for each
nullable column.

### 02-containers.md

Container model:

| container | notes |
|---|---|
| `node_block` | N nodes with the same label; columnar |
| `edge_block` | N edges with the same label; columnar; src+dst as node_id columns |
| `graph` | a sequence of node_blocks and edge_blocks |

**node_block** mandatory columns (col_ids 0 and 1):
- col_id 0: `nid` (uint64) — node identifiers, monotone, delta-coded
- col_id 1: `label` (string, nullable) — node label; NULL means no label

User-defined property columns use col_ids ≥ 2.

**edge_block** mandatory columns (col_ids 0–3):
- col_id 0: `eid` (uint64) — edge identifiers, monotone, delta-coded
- col_id 1: `src` (node_id) — source node nid
- col_id 2: `dst` (node_id) — destination node nid
- col_id 3: `label` (string, nullable) — edge label; NULL means no label

User-defined property columns use col_ids ≥ 4.

Directionality: edges are **directed by default** (src → dst).
Undirected graphs are an application-level convention: the encoder
emits each undirected edge once with src ≤ dst (canonical form) and
the application treats src/dst as interchangeable.

Label interning: node and edge labels are UTF-8 strings stored using
the core strmap mechanism (same as weavepack-json string interning).
Label values that repeat across blocks are deduplicated at the strmap
level, not separately.

### 03-paths.md

Path grammar for delta operations:

- `node/N` — a node by nid (uint64 integer in decimal)
- `edge/E` — an edge by eid (uint64 integer in decimal)
- `node/N.prop_name` — a property of a node, addressed by name
- `node/N[col_id]` — a property of a node, addressed by col_id integer
- `edge/E.prop_name` — a property of an edge, addressed by name
- `edge/E[col_id]` — a property of an edge, addressed by col_id integer
- `@nid` — the mandatory nid column (col_id 0 of node_block)
- `@eid` — the mandatory eid column (col_id 0 of edge_block)
- `@src` — the mandatory src column (col_id 1 of edge_block)
- `@dst` — the mandatory dst column (col_id 2 of edge_block)
- `@label` — the mandatory label column (col_id 1 of node_block; col_id 3 of edge_block)

Schema-addressed paths (schema required):
- `.NodeLabel` — all nodes with a given label
- `.NodeLabel.prop_name` — all nodes of a label, addressing a property column
- `-EdgeLabel` — all edges with a given label (dash prefix to distinguish from node)
- `-EdgeLabel.prop_name` — all edges of a label, addressing a property column

### 04-deltas.md

Delta operations:

| Op | Code | Description |
|---|---|---|
| `node_insert` | 0 | Insert N nodes into the graph (columnar) |
| `node_delete` | 1 | Remove nodes by nid (and implicitly incident edges) |
| `edge_insert` | 2 | Insert N edges into the graph (columnar) |
| `edge_delete` | 3 | Remove edges by eid |
| `prop_set` | 4 | Update a property value for a node or edge |
| `subgraph_replace` | 5 | Atomically replace a named subgraph |

Codes 6–7 are reserved. A decoder reading an unknown op code MUST
refuse with `unknown_delta_op`.

Op semantics are detailed in spec doc 04-deltas.md.

### 05-conformance.md

Test corpus structure:

- **Level 1** (block only): encode + decode a node_block and edge_block.
  Round-trip. Covers all ctypes including node_id (ctype 15).
- **Level 2** (single delta): node_insert or edge_insert + one read-back.
- **Level 3** (delta chain): multi-frame chain, all 6 op types.

### 06-schemas.md

Schema sidecar (parallel to tabular and log 06-schemas.md):

- Declares node labels with their column schemas
- Declares edge labels with their column schemas
- Schema format: canonical JSON (sorted keys, no trailing decimals)
- Schema hash: SHA-256 of canonical JSON bytes
- Graph schema format includes RDF-mode flag:
  - When `rdf_mode: true`, nid values are URI strings interned in
    strmap; the encoder automatically interns subject/object URIs
    and emits a node_block with a `uri` string column alongside `nid`.
    This mode requires the schema to declare which labels are IRI nodes
    vs blank nodes.

### 07-benchmarks.md

Benchmark methodology and results vs. GraphML + gzip.

Target scenarios:

1. **Social graph snapshot**: 10 000 nodes, 100 000 edges. Node schema:
   (user_id uint64, name string, country string, joined_date date32).
   Edge schema: (follows, no extra properties). Compare
   node_block + edge_block size vs. GraphML + gzip and RDF/Turtle + gzip.
   Gate: weavepack-graph within 2× of GraphML + gzip on snapshot
   (gzip has strong text back-reference advantage on structured XML).

2. **Incremental edge stream**: 1 000 `edge_insert` delta frames, each
   adding 100 edges to the social graph. Compare: weavepack stream
   (init + 1 000 delta frames) vs. per-snapshot GraphML + gzip.
   Gate: weavepack ≥10× smaller than per-snapshot GraphML + gzip.

3. **Mixed node/edge updates (CDC-style)**: 500 frames mixing
   node_insert, edge_insert, edge_delete, prop_set. Compare
   stream size vs. JSON-LD per-snapshot + gzip.
   Gate: weavepack ≥5× smaller than per-snapshot JSON-LD + gzip.

## Implementation plan

### Step G.1 — Spec docs 00–04 (this cycle)

Write spec docs 00-overview.md through 04-deltas.md in
`weavepack/profiles/graph/`. No implementation yet; spec first.

### Step G.2 — JS reference implementation

`sdk/src/profiles/graph/` — parallel to `sdk/src/profiles/log/`.

- `types.js` — ctype constants including node_id (ctype 15)
- `encoder.js` — node_block + edge_block encoder; delta encoder
- `decoder.js` — node_block + edge_block decoder; delta decoder
- `apply.js` — applyChain for graph delta ops
- `index.js` — exports `encodeGraph`, `decodeGraph`, `encodeChain`, `decodeChain`

All existing tests must continue to pass (profile isolation).

### Step G.3 — Conformance test corpus

`weavepack/profiles/graph/test-vectors/`:

- `types/scalars.json` — round-trips for all ctypes including node_id
- `types/nulls.json` — nullable properties, null bitmaps
- `containers/node_blocks.json` — multi-property node blocks
- `containers/edge_blocks.json` — directed edges with src/dst
- `containers/graph.json` — combined node + edge document
- `deltas/node_insert.json` — node append chains
- `deltas/node_delete.json` — node removal with implicit edge cleanup
- `deltas/edge_insert.json` — edge append chains
- `deltas/edge_delete.json` — edge removal
- `deltas/prop_set.json` — property update ops
- `schemas/schemaful.json` — schema-addressed paths and label dispatch

Extend `weavepack/tools/verify-test-vectors.js` with a graph profile handler.

### Step G.4 — Rust crate

`impl/rust/weavepack-graph/` — parallel to `impl/rust/weavepack-log/`.

### Step G.5 — Python implementation

`impl/python/weavepack_graph/` — parallel to `impl/python/weavepack_log/`.

`weavepack/tools/benchmark-graph.js` and

## Benchmark gate (v0.6 ship criterion)

v0.6 ships when all of these hold:

1. weavepack-graph spec docs 00–07 are written and consistent.
2. JS + Rust + Python implementations pass all conformance vectors.
3. Benchmark scenario 2 (incremental edge stream): chain is ≥10× smaller
   than per-snapshot GraphML + gzip for 1 000 edge_insert frames of
   100 edges each.
4. Benchmark scenario 3 (mixed CDC updates): chain is ≥5× smaller than
   per-snapshot JSON-LD + gzip for 500 mixed-op frames.
5. Benchmark scenario 1 (single snapshot): within 2× of GraphML + gzip
   for 10 000 nodes + 100 000 edges (gzip has strong text back-reference
   advantage on XML; column encoding wins on the delta side).
