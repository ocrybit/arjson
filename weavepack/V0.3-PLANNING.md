# Weavepack v0.3 ‚Äî planning document

**Status:** Planning. Defines the scope for the v0.3 cycle.
**Precondition:** v0.2 complete (2026-05-08). All v0.2 Tier 1 items shipped.

## What v0.3 is

Two parallel threads:

1. **Profile #3: weavepack-wire** ‚Äî the primary deliverable. A
   protobuf-equivalent RPC-message format with native delta chains.
   No other binary RPC format has per-update deltas; this is the gap.

2. **Deferred v0.2 items that gained traction** ‚Äî see ¬ßCarry-forward
   below. Treated as secondary; they unblock or improve weavepack-wire
   without distracting from the main goal.

## Why weavepack-wire

The ROADMAP ranks profiles by strategic value:

1. weavepack-tensor ‚úì shipped
2. **weavepack-wire** ‚Äî bigger market, strongest delta story
3. weavepack-tabular ‚Äî parquet-adjacent, useful but crowded

weavepack-wire's unique value proposition against protobuf v3 + brotli:

| | bit-pack | deltas | schema | self-desc | streaming |
|---|---|---|---|---|---|
| protobuf v3 | ‚úó (byte TLV) | ‚úó | ‚úì | ‚úó | ‚úì |
| **weavepack-wire** | **‚úì** | **‚úì** | **‚úì** | **‚úì** | **‚úì** |

The win is sharpest for **incremental API responses** (e.g., a game-state
diff, a streaming LLM token block, a database cursor update). These
are payloads that protobuf clients serialize as full snapshots because
protobuf has no delta primitive; weavepack-wire emits a delta chain.

Estimated compression win over protobuf+brotli for incremental payloads:
same order of magnitude as the tensor 3‚Äì4√ó single-element-delta win.
Snapshot parity expected within 5‚Äì10% (bit-pack advantage vs. TLV).

## Profile #3 spec plan

Parallel structure to the tensor profile. All documents live under
`weavepack/profiles/wire/`.

### 00-overview.md

Context, motivation, incumbent comparison, open design questions.

### 01-types.md

Value types (analogous to protobuf scalar types):

| type | tag | bits | notes |
|---|---|---|---|
| `bool` | 0 | 1 | |
| `int32` | 1 | 32 | zigzag encoded for signed efficiency |
| `int64` | 2 | 64 | zigzag encoded |
| `uint32` | 3 | 32 | |
| `uint64` | 4 | 64 | |
| `sint32` | 5 | 32 | pre-zigzagged (caller-managed sign) |
| `sint64` | 6 | 64 | pre-zigzagged |
| `float32` | 7 | 32 | IEEE 754 binary32, LE |
| `float64` | 8 | 64 | IEEE 754 binary64, LE |
| `string` | 9 | variable | LEB128-length + UTF-8 bytes |
| `bytes` | 10 | variable | LEB128-length + raw octets |
| `enum` | 11 | variable | int32 with schema-declared name set |
| reserved | 12‚Äì15 | ‚Äî | for future extension |

No `null` type; absent optional fields use the field-absence encoding
from 02-containers.md. Presence is structural, not typed.

### 02-containers.md

Container types:

| container | tag | notes |
|---|---|---|
| `message` | 0 | ordered set of field_number‚Üívalue pairs |
| `repeated` | 1 | variable-length sequence of a single value type |
| `map` | 2 | key‚Üívalue pairs where key is string or integer |
| `oneof` | 3 | exactly one field from a declared set is present |

Field numbers are uint32 (same as protobuf). Field names are
schema-optional; without a schema, field numbers are the only identity.
Message layout in the wire format: field_numbers column + value columns.
Absence of an optional field: field_number not present in the snapshot
columns (field_number list is sparse; gaps = absent). Required fields:
declared in the schema (similar to protobuf required, but not enforced
at wire level ‚Äî schema validators enforce).

### 03-paths.md

Path grammar for addressing fields in delta operations.

- `.field_name` or `[field_number]` for message fields
- `[n]` for repeated-field indices
- `{key}` for map entries (key is string or integer)
- `.oneof_field_name` for oneof cases (only one active at a time)

Examples:
- `[1]` ‚Üí field number 1 of the root message
- `.name` ‚Üí field named "name" (schema required for name lookup)
- `[2][0]` ‚Üí first element of the repeated field #2
- `[3]{user_id_123}` ‚Üí map entry with key "user_id_123" in field #3
- `.payload.oneof_response.success_body` ‚Üí nested path

### 04-deltas.md

Delta operations (parallel to tensor 04-deltas.md):

| op | code | description |
|---|---|---|
| `field_set` | 0 | set (or replace) a single scalar field at path |
| `field_delete` | 1 | remove an optional field at path |
| `message_replace` | 2 | replace an entire sub-message |
| `repeated_append` | 3 | append N elements to a repeated field |
| `repeated_splice` | 4 | delete and/or insert at a position |
| `map_set` | 5 | set or replace a map entry |
| `map_delete` | 6 | remove a map entry |
| `oneof_switch` | 7 | change the active oneof case |

Delta composition laws (same as core 05-deltas.md, instantiated for wire):

- `apply(field_set(p, v2), apply(field_set(p, v1), m)) = apply(field_set(p, v2), m)` ‚Äî last-write-wins
- `apply(field_delete(p), apply(field_set(p, v), m)) = m[without p]` ‚Äî delete after set = absent
- `apply(field_set(p, v), apply(field_delete(p), m)) = m[with p=v]` ‚Äî set after delete = present
- `repeated_splice` composition follows the same rules as JSON splice

### 05-conformance.md

Test corpus structure (parallel to tensor 05-conformance.md).

Levels:

- **Level 1** (snapshot only): encode + decode a wire snapshot.
  No delta ops.
- **Level 2** (single delta): encode a snapshot, apply one delta,
  verify final state. Round-trip invariant must hold.
- **Level 3** (delta chain): multi-frame chain, all 8 op types.

### 06-schemas.md

Schema sidecar for weavepack-wire (parallel to tensor 06-schemas.md).

The schema declares:
- Message type names and field numbers ‚Üí names
- Field types (value type or nested message type)
- Required vs. optional vs. repeated vs. map vs. oneof
- Enum value sets

Schema format: JSON (same canonical form as tensor schema ‚Äî sorted keys,
no trailing decimals, no scientific notation). Schema hash: SHA-256 of
the canonical JSON bytes, same algorithm as tensor.

Without a schema:
- Field names are unavailable (only field numbers usable in paths)
- Type information is inferred from encoded values (same as protobuf
  in self-describing mode, except weavepack-wire carries vtype columns)
- Enum values are raw integers

With a schema:
- Field numbers are hidden; paths use field names
- Type validation at decode time
- Enum symbolic names available

### 07-benchmarks.md

Benchmark methodology and results vs. protobuf v3 + brotli.

Target scenarios:

1. **Full snapshot**: a 10-field message, 5 nested messages, one repeated
   field of 100 elements. Compare snapshot size vs. protobuf+brotli.

2. **Incremental API response**: a game-state message, 1000 update steps,
   each step changes 1‚Äì5% of fields. Compare chain size vs. (protobuf
   snapshot √ó 1000) + brotli.

3. **Streaming token stream**: repeated field of string tokens, appended
   10 at a time, 100 append steps. Compare chain size vs. (protobuf
   snapshot √ó 100) + brotli.

Gate: weavepack-wire beats protobuf+brotli by ‚â•2√ó on scenario 2 and
scenario 3 (where delta chains provide structural advantage). Scenario 1
should be within 10% (bit-pack advantage vs. TLV overhead roughly cancel
for typical message sizes).

## Carry-forward deferred items from v0.2

These items were deferred in v0.2 and are still open. Some are actionable
as part of the wire profile work; others remain research-grade.

### CF.1 C.1 ‚Äî JSON schema sidecar

"Only if benchmarks justify." Carry to v0.3, still conditional.
JSON's strmap dedup already handles repeated-key efficiency. A JSON
schema sidecar would only help if (a) the value types are highly
repetitive and (b) the caller has a stable schema. If a real consumer
with a stable JSON schema appears, revisit and benchmark.

**Verdict: deferred to v0.4+ pending evidence.**

### CF.2 C.2 ‚Äî Schema-as-delta-chain

Research-grade. Requires designing a recursive meta-schema language
and the schema-evolution protocol. Relevant for long-lived databases
where schema versions change over decades (Avro's main use case).

This becomes concrete if a production adopter hits schema evolution
needs. Until then, the current "schema hash changes = provide new schema"
is sufficient.

**Verdict: deferred to v0.4+ pending adopter demand.**

### CF.3 C.5 ‚Äî Cross-architecture endianness validation

Needs a big-endian CI runner (Linux ppc64 or s390x). The current CI
(`.github/workflows/conformance.yml`) runs on `ubuntu-latest` which
is LE. The fix is a matrix entry `runs-on: ubuntu-latest` with
`QEMU_CPU=ppc64`.

**Action (v0.3 optional):** Add a BE matrix entry to the CI workflow.
Low risk (pure test coverage), no implementation changes needed.

### CF.4 D.5 ‚Äî Independent third-party implementation

Open invitation per `weavepack/governance/07-implementors-guide.md`.
Cannot do ourselves; requires an external contributor. The implementors
guide is the mechanism ‚Äî if someone writes a Go or Zig implementation
from spec, that validates both the spec quality and the conformance
corpus.

**Verdict: tracked in registry doc; no action from us.**

### CF.5 B.2 ‚Äî remove-key in array elements

The `diffArray` implementation safely falls back to full-element replace
for remove-key operations within array-element objects (the vref ordering
constraint). The safety boundary is correct and documented. Lifting it
requires a redesign of the vref merge algorithm.

If a production user hits the "I'm removing keys from objects inside
large arrays and the delta is too large" wall, that's the signal to
redesign. Until then: safe boundary, documented, no action.

**Verdict: deferred to v0.4+ pending adopter pressure.**

## What v0.3 does NOT include

- **weavepack-tabular**: deferred to v0.4. Wire profile first, tabular second.
  Rationale: tabular needs row-level delta design which benefits from lessons
  learned in the wire profile's message-level delta design.
- **Formal verification** (Lean/Coq): out of scope as per ROADMAP.md.
- **Network transport layer**: weavepack defines bytes, not how they're sent.
- **IETF standardization**: premature; do it after ‚â•2 independent certified impls.
