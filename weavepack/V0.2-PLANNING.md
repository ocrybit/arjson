Citation: `weavepack/profiles/tensor/04-deltas.md`.

`region_replace` is the more important of the two — it covers the
dense-block-update case (e.g. updating one row of a matrix). The
heuristic in 04-deltas.md says: "If changed elements are < 50% of
bounding box: emit element_set; otherwise region_replace." Without
region_replace, dense updates fall back to full `tensor_replace`,
which is wasteful.

`quant_change` is rarer; only relevant for QAT (quantization-aware
training) pipelines that periodically re-quantize.

### A.3 Delta-from-prior arithmetic compression — ✓ DONE

**Status: ✓ decoder + encoder shipped (JS + Rust + Python decoders;
JS + Rust encoder heuristic). Rust mode=1 encoder complete (2026-05-07).**

Mode bit infrastructure landed in `sdk/src/profiles/tensor/index.js`:
- Encoder emits mode=0 or mode=1 for every `tensor_replace` based
  on max-abs-delta heuristic (threshold 0.01; fp32/fp64 only).
- Decoder handles mode=0 (absolute, previous behaviour) and mode=1
  (per-element arithmetic delta: `new[i] = base[i] + delta[i]`).
- All dtypes supported: fp32, fp64, int8/16/32, uint8/16/32, int64,
  uint64. fp16/bf16 handled via f32 intermediary per RFC 0001.
- 3 new corpus vectors in `deltas/delta_from_prior/fp32_and_int32.json`
  (raw-delta format; tested via `delta_bytes_hex` field in verifier).
- 2 new corpus vectors in `deltas/delta_from_prior/fp64.json`
  (full-chain format; encoder naturally emits mode=1 for fp64 when
  max abs delta ≤ 0.01; tests encoder + decoder + round-trip; 196/196
  total conformance vectors, was 194/194).
- 3 new SDK unit tests in `tensor-profile.test.js` (fp32 round-trip,
  int32 with wrapping, bit-flip technique).

Empirical end-to-end win (brotli-stacking.js dense scenario, 100
training steps over 1024-element fp32 with ±0.001 noise):
- weavepack+brotli with mode=1 heuristic: 1.6× smaller than
  safetensors+brotli (was ~1.0× with mode=0 only).

A sub-mode of `tensor_replace`: instead of emitting absolute new
values, emit per-element deltas `new_val - base_val`. For training
deltas where most values change by a small amount, the deltas have
many leading zero bits and compress beautifully.

Citation: `weavepack/profiles/tensor/04-deltas.md`, "Compression
beyond delta" section.

**Empirical measurement** (1024 fp32 tensor, every element changes
by ±magnitude; brotli win = mode=0 brotli ÷ mode=1 brotli):

| magnitude | mode=0 + brotli | mode=1 + brotli | brotli win |
|-----------|-----------------|-----------------|------------|
| ±0.0001   | 3791 B          | 2332 B          | **1.63×**  |
| ±0.001    | 3767 B          | 2765 B          | 1.36×      |
| ±0.01     | 3813 B          | 3383 B          | 1.13×      |
| ±0.1      | 3777 B          | 3673 B          | 1.03×      |
| ±1.0      | 3844 B          | 3853 B          | 1.00×      |

The win shrinks as update magnitude grows (deltas approach
absolute values). This sweep gives the encoder heuristic a
concrete decision boundary: emit mode=1 when typical per-element
change magnitude is small (< ~0.01). Above that, mode=0 is fine.

Reproducer:
`weavepack/profiles/tensor/examples/delta-from-prior-mode-bit.js`

### A.4 Sub-tensor random access (skip-load) — ✓ COMPLETE

**Status: ✓ Implemented in JS + Rust + Python (no wire format change).**

Because the schema gives each tensor's exact `dataBytes(dtype, shape)`,
the decoder can compute each tensor's bit-offset arithmetically and seek
directly to tensor N without reading tensors 0..N-1.

Two new exported functions in `sdk/src/profiles/tensor/index.js`:

- `listTensorsSchemaful(bytes, registry)` → `string[]`
  Returns the canonical tensor names without decoding any data.
- `decodeTensorSchemaful(bytes, name, registry)` → `{ dtype, shape, data }`
  Seeks to the named tensor and decodes only its data block.
  Handles dequantization (QINT8/QINT4/QFP8) identically to the
  full `decodeDocumentSchemaful`.

No wire format change: the schemaful layout already stores data blocks
consecutively with sizes derivable from the schema. Skip calculation:
bit offset to tensor k = 258 (2-bit discriminant + 256-bit hash) +
Σ dataBytes(dtype_i, shape_i)×8 for i in 0..k-1.

11 unit tests in `sdk/test/tensor-skip-load.test.js`:
- first / middle / last tensor, all-tensors loop, single-tensor doc,
  mixed dtypes, qint8 dequantization, error on unknown name, byte-offset
  arithmetic cross-check.

Rust: `list_tensors_schemaful` and `decode_tensor_schemaful` in decode.rs;
shared `parse_schemaful_header` helper; 7 inline unit tests.

Python: `list_tensors_schemaful`, `decode_tensor_schemaful` in
`impl/python/weavepack_tensor/decoder.py`. Uses `_parse_schemaful_header`
helper (shared with A.5). Sets `_BitReader.bit_pos` directly to skip
preceding tensors. 11 unit tests in `impl/python/test_skip_load.py`
(first/middle/last tensor, parity with full decode, unknown name error,
qint8 dequantization, byte-offset cross-check, 5-tensor doc).

Citation: `weavepack/profiles/tensor/00-overview.md`, "Open questions" #3.

### A.5 Streaming mode — ✓ COMPLETE (JS + Rust + Python)

**Status: ✓ Implemented in JS + Rust + Python (no wire format change).**

New exported generator in `sdk/src/profiles/tensor/index.js`:

- `iterateTensorsSchemaful(bytes, registry)` → `Generator<{ name, dtype, shape, data }>`
  Yields one tensor at a time in canonical (sorted) name order.
  Maintains a single sequential bit-position cursor — no per-tensor offset
  arithmetic, no seeking backwards. Identical materialize + dequantize
  logic to `decodeDocumentSchemaful`; yielded values are byte-exact.

Key properties vs A.4:
- A.4 (`decodeTensorSchemaful`): random access — seek to any named tensor in
  a single call; O(n) offset arithmetic to skip preceding tensors.
- A.5 (`iterateTensorsSchemaful`): sequential streaming — iterate all tensors
  in one pass; cursor advances naturally without recomputing offsets.
- Lazy: stopping iteration early (breaking out of the for-of loop) does not
  throw and does not read remaining tensor data.

9 unit tests in `sdk/test/tensor-stream.test.js`:
- canonical order, full-decode parity, single-tensor doc, mixed dtypes,
  qint8 dequantization, five-tensor doc, early-stop laziness,
  A.4 cross-check (streaming == skip-load for same tensor), schemaless
  payload error.

SDK: 2279 tests pass (was 2268; +9 streaming tests; +2 B.2 array-of-objects delta tests).

Rust: `SchemafulIter<'bytes, 'reg>` struct implementing `Iterator`; 7 unit tests.

Python: `iterate_tensors_schemaful` generator in
`impl/python/weavepack_tensor/decoder.py`. Shares `_parse_schemaful_header`
and `_dequantize` helpers with A.4. Advances a single `_BitReader` cursor
without seeking; early `gen.close()` does not raise. 9 unit tests in
`impl/python/test_skip_load.py` (canonical order, full-decode parity,
single-tensor doc, mixed dtypes, early stop, A.4 cross-check, qint8
dequantization, schemaless-rejection, 5-tensor doc).

Citation: `weavepack/profiles/tensor/00-overview.md`, A.5 note.

## B. JSON profile (weavepack-json)

### B.1 Move/copy ops

JSON Patch (RFC 6902) defines `move` and `copy` operations for
relocating values within a tree. The JSON profile rejects these by
design ("no conformance theater"), but a future revision could add
them under a different op name if the wire-format savings are real.

Citation: `weavepack/profiles/json/04-deltas.md`, "Known limitations"
#1 + ROADMAP.md design philosophy.

The argument against: move/copy are syntactic sugar for
remove + add; they don't save bits. Reject unless someone
demonstrates a measurable win.

### B.2 Splice on object arrays (per-element diff) — ✓ IMPLEMENTED

**Status: ✓ Implemented (2026-05-07, extended 2026-05-07). Primitive-leaf
replacement and add-key-to-non-empty-element both fully working. Remove-key
within array elements still falls back to reanchor (safe boundary).**

`diffArray` in `sdk/src/profiles/json/diff.js` now emits sub-path ops for
array-of-objects when primitive leaf values change within existing elements.
Previously it bailed to full-array replace whenever any element was non-primitive.

Fix 1 (`diff.js`): two-pass approach:
- First pass: for each modified element, if both old and new are plain objects,
  recurse into `diff(a[i], b[i], path[i])` and check whether ALL sub-ops are
  safe: (a) "replace"/"diff" always; (b) "add" only when `a[i]` already has at
  least one key (so the element is anchored in the merged vref table). "remove"
  is never allowed via sub-op. If any sub-op fails the check, bail to replaceOp.
- Second pass: emit the collected sub-ops instead of the whole-array replace.

Fix 2 (`builder.js`): in `handleTerminalKey` for `ctype===0 && ntype===1`
(descend into empty object under an array slot), added `set(k2, json.length)`
before `json.push({})` so the builder's `init1` map records the position of
the newly created empty object. Without this, re-entrant vrefs for the same
element would push a duplicate `{}`.

Fix 3 (`artable.js`): in `delta()`, when adding a new string key under an
existing path, compute `parentContainerKref` correctly. For map-type (numeric)
containers the container kref is `i` (not `i+1`); for str-type containers it
is `i+1`; for root it is `1`.

Fix 4 (`index.js`): `update()` now null-checks `artable.delta()` return and
reanchors on null, rather than crashing.

Scope: "add" for a NEW key is supported when `a[i]` has existing keys (safe
because those keys anchor the element's position in the vref table). "remove"
always falls back to reanchor — delete markers appended to the end of the merged
vref table can displace later elements during builder reconstruction.

Citation: `weavepack/profiles/json/04-deltas.md`, splice section.

### B.3 Range delete for object keys

Bulk-delete primitives don't exist for objects (object keys have
no positional ordering, so no equivalent to array splice). A
future revision could add a "delete keys matching predicate" op,
but the protocol-side savings are unclear vs N individual remove
ops.

## C. Core protocol (weavepack-core)

### C.1 Schema sidecar protocol

`weavepack/core/06-schemas.md` defines the *concept* but most
details (schema language, hash function, sidecar fetching) are
profile-specific. The tensor profile has implemented schemaful
encoding (5.5 done); JSON has not. Whether JSON benefits from
schemas is an open question — JSON's strmap dedup already covers
the most common case (repeated keys).

### C.2 Schema-as-delta-chain

The mention of recursive meta-schemas (a schema that's itself a
weavepack delta chain). Cited as "the most ambitious version of
the schema feature" in core/06-schemas.md. Deferred to Phase 5+.

Concrete use case: long-lived database tables where the schema
itself evolves over time, but old payloads must remain decodable.
This is what Avro tries to do; weavepack could do it cleaner via
delta chains.

### C.3 Explicit profile-id field — ✓ COMPLETE (RFC 0002 Accepted + Implemented)

**Status: ✓ RFC 0002 Accepted (2026-05-07). JS implementation shipped.**

`sdk/src/dispatch.js`: `wrapPayload(bytes, profileId)` prepends the 4-byte
v1.2 magic header `[0x57, 0x50, 0x12, pid]`; `peekHeader(bytes)` detects and
strips it (returns null for v1.x payloads — the v1.x JSON fallback path).
`PID.JSON = 0x00`, `PID.TENSOR = 0x01`. Exported from `sdk/src/index.js`.
Conformance vectors: 4 JSON + 2 tensor v1.2 vectors in `test-vectors/v1.2/`.
19 unit tests in `sdk/test/dispatch.test.js`. 202/202 conformance vectors pass.

Citation: `weavepack/rfcs/0002-explicit-profile-id.md`.

### C.4 Streaming-mode conformance

The conformance corpus assumes whole-payload encode/decode.
Streaming-mode (incremental decode of a large payload) needs a
separate vector category. Currently not specified.

Citation: `weavepack/core/09-conformance.md`, Open issues #2.

### C.5 Cross-architecture endianness validation

Spec mandates little-endian for multi-byte fields. All current
implementations and test machines are LE. Big-endian validation
needs a CI runner on a BE host (Linux ppc64, s390x).

Citation: `weavepack/profiles/tensor/05-conformance.md`, Open
issues #1.

Citation: `impl/rust/weavepack-json/src/struct_encode.rs`.

### D.2 Python JSON structured-mode — ✓ COMPLETE

**Status: 93/93 conformance vectors pass. Full snapshot + delta chain
decoder shipped.**

Delta chain decoder implemented in
`impl/python/weavepack_json/decoder.py`:
- `parse_chain()`: LEB128-framed payload parser.
- `_decode_payload()`: context-aware structured-mode decoder for both
  snapshot (is_delta=False) and delta (is_delta=True) payloads.
  Delta mode uses `bits(base_krefs_len+2)` initial cbits and omits
  the root ktype/key entry.
- `_apply_delta_to_json()`: applies delta operations (set, delete,
  splice_replace, splice_delete, strdiff) directly to the live JSON
  tree without ARTable intermediate state.
- `_apply_strdiff()`: decodes and applies byte-oriented fast-diff ops
  for efficient string patch operations.
- `decode_chain()`: top-level API; decodes snapshot then chains
  incremental deltas to produce the final JSON value.
- 93/93 conformance vectors pass (0 skips); zero regressions in the
  JS 2188-test suite and 151-vector JS conformance corpus.

Zero-extend-past-end fix in `_BitReader.read_bits()` matches the JS
decoder's behaviour for payloads padded to byte boundaries.
splice_rep value-column dispatch (`_vt_num_tag`, `_vt_str_type`,
`_vt_is_bool`) corrected to handle the inner-vtype field properly.

Citation: `impl/python/weavepack_json/decoder.py` + conformance
output.

### D.3 Python JSON structured-mode encoder — ✓ COMPLETE

**Status: 68/68 encoder vectors byte-exact. Full encode parity with Rust.**

Full ARTable structured-mode encoder implemented in
`impl/python/weavepack_json/encoder.py`:
- `_ColBuf`: column accumulator (list of (val, n_bits) pairs replayed at
  finalize, avoiding byte-padding between columns).
- `_StructEnc`: direct port of Rust `StructEncoder` in
  `impl/rust/weavepack-json/src/struct_encode.rs`.
  - vlink/klink diff+RLE: adaptive bit-width (prev_bits/prev_kbits),
    diff encoding for small deltas, RLE runs (≥4 identical encoded values).
  - vtypes RLE: run-length encoding for consecutive identical vtypes (≥4).
  - nums diff+RLE: prev_num tracking, diff for small deltas, RLE (≥3).
  - Flag columns (vflags, kflags, bools): 2-bit prefix (all-zero/all-one/mixed).
  - Strmap: shared key+value intern map; keys use ktype=2 + keylen=0 for
    dedup refs; values use short(0) + flag=0 + short(idx).
- `encode()` now routes non-empty arrays/objects to `_encode_structured()`.

Python JSON conformance: 93/93 decode + 68/68 encode (was 37/68 before).
Cross-language encoder conformance: JS 68/68, Rust 68/68, Python 68/68.

### D.4 Python tensor delta encoder — ✓ COMPLETE

**Status: compute_delta + encode_delta implemented; 14/14 corpus delta
vectors byte-exact against JS reference; 15 unit tests.**

`impl/python/weavepack_tensor/encoder.py` now exports:
- `compute_delta(base_doc, new_doc)` → list of op dicts (all 6 ops:
  tensor_replace, tensor_add, tensor_remove, element_set, region_replace,
  quant_change).
- `encode_delta(base_doc, new_doc)` → bytes or None if identical.
  Same heuristics as JS/Rust: mode=1 for fp32/fp64 when max abs delta ≤ 0.01;
  element_set for sparsity < 30%; region_replace when bbox density > 50%.
  Also emits quant_change for QINT8/QINT4/QFP8 tensors with changed scale or
  zero_point.

`impl/python/weavepack_tensor/__init__.py` exports `compute_delta` and
`encode_delta`. `impl/python/test_encode_delta.py`: 15 unit tests
(5 corpus round-trips, 10 unit cases).

### D.6 Rust JSON delta chain decoder — ✓ COMPLETE

**Status: parse_chain + decode_chain implemented in Rust; 93/93 conformance with
25 vectors now verified at chain level (expected_chain_bytes_hex → expected_final).**

New functions in `impl/rust/weavepack-json/src/decode.rs`:
- `parse_chain(data)` — LEB128-framed payload splitter.
- `decode_snapshot_for_chain(data)` → `(Value, ChainContext)` — snapshot decode
  that also captures krefs/keys/strmap for use by the next delta in the chain.
- `decode_delta_cols(r, base_ctx)` — delta payload decode using base context;
  `initial_cbits = bits(base_krefs_len + 2)` when krefs non-empty; combined
  krefs/keys; delta strmap = base + new literal string values.
- `apply_delta_ops(root, delta)` — applies all ops (set/delete/splice_del/
  splice_rep/strdiff) via recursive `nav_apply_delta` path traversal.
- `apply_strdiff(original, data, strmap)` — string diff application (delete +
  insert ops at character positions, sorted by position).
- `decode_chain(data)` — public driver: parse frames, decode first as snapshot,
  apply each subsequent frame as delta.

Also fixed a latent bug: `vt_num_tag` now handles `SpliceReplace{typ:4/5/6}`,
so splice_rep values with integer/float payloads are correctly read into the
nums array. This was harmless for existing snapshot tests (no SpliceReplace in
snapshot mode) but caused "nums exhausted" for delta chain decoding.

Conformance binary extended: `run_delta` verifies `expected_chain_bytes_hex` →
`expected_final` for all 25 JSON chain vectors (18 multi-frame + 7 single-frame).

### D.5 Independent third-party implementation

All current impls are derived from the JS reference. A genuine
third-party impl (someone else writing from spec without consulting
JS source) would validate the spec is implementable independently.

The Python PoC partially does this (was written from prose only),
but is by the same maintainer. A truly independent third-party
impl in any language remains an open invitation — see
`weavepack/governance/02-profile-registry.md`.

## Process

This planning doc gets revised as features ship. When a Tier 1
item lands, mark it in this file (or remove if it's truly done).
When new deferrals are recorded in spec docs, add them here so
this stays the single source of truth for "what's deferred".

Update cadence: refresh after each major milestone (e.g. RFC
acceptance, new phase completion).
