# Weavepack v0.2 — deferred features inventory

**Status:** Planning. Catalogues features deferred from v0.1 across
the spec. Not a roadmap — that's `ROADMAP.md`. This is a
working list for prioritization when v0.1 is shipped.

## Source extraction

For each deferred item, the citation is the spec doc + section
where the deferral was recorded.

## A. Tensor profile (weavepack-tensor)

### A.1 Additional dtypes

| dtype | Tag | Status | RFC |
|---|---|---|---|
| `int4` / `uint4` | 1 / 2 | Spec'd, unimplemented | (no RFC yet) |
| `fp8e4m3` / `fp8e5m2` | 11 / 12 | Spec'd, unimplemented | (no RFC yet) |
| `cfloat32` / `cfloat64` | 17 / 18 | Spec'd, unimplemented | (no RFC yet) |
| `qint4` / `qint8` / `qfp8` | 28-30 | Spec'd, unimplemented | (no RFC yet) |
| `fp16` / `bf16` | 13 / 14 | RFC 0001 in Discussion | RFC 0001 |

Citation: `weavepack/profiles/tensor/01-types.md` (dtype registry).

Estimated effort:
- int4/uint4: small (sub-byte packing already done for bool;
  generalize to nibble pack)
- fp8e4m3/fp8e5m2: medium (no native JS/Rust types; bit-level
  conversion code per format)
- cfloat32/64: small (interleave existing fp32/fp64 pairs)
- qint4/qint8/qfp8: medium-large (requires per-tensor scale +
  zero_point in the schema; encoder/decoder must apply
  `value = (qval - zp) * scale` on dequantize)

### A.2 Additional delta operations

| op | Code | Status |
|---|---|---|
| `tensor_replace` | 0 | ✓ implemented |
| `tensor_add` | 1 | ✓ implemented |
| `tensor_remove` | 2 | ✓ implemented |
| `region_replace` | 3 | ✓ JS encoder + decoder; ✓ Rust encoder + decoder; ✓ Python decoder. 4 corpus vectors landed; cross-language byte-equivalence verified. |
| `element_set` | 4 | ✓ implemented |
| `quant_change` | 5 | Spec'd, unimplemented |

Citation: `weavepack/profiles/tensor/04-deltas.md`.

`region_replace` is the more important of the two — it covers the
dense-block-update case (e.g. updating one row of a matrix). The
heuristic in 04-deltas.md says: "If changed elements are < 50% of
bounding box: emit element_set; otherwise region_replace." Without
region_replace, dense updates fall back to full `tensor_replace`,
which is wasteful.

`quant_change` is rarer; only relevant for QAT (quantization-aware
training) pipelines that periodically re-quantize.

### A.3 Delta-from-prior arithmetic compression

A sub-mode of `tensor_replace`: instead of emitting absolute new
values, emit per-element deltas `new_val - base_val`. For training
deltas where most values change by a small amount, the deltas have
many leading zero bits and compress beautifully.

Citation: `weavepack/profiles/tensor/04-deltas.md`, "Compression
beyond delta" section.

Estimated savings: 5-10× on top of element_set for typical training
workloads. This is the unique-to-weavepack win that justifies the
ML-checkpoint use case.

### A.4 Sub-tensor random access (skip-load)

Currently the wire format requires reading sequentially through all
tensors to find any one. A future revision could add an offset table
in the schemaful header so consumers can seek directly to tensor N.

Citation: `weavepack/profiles/tensor/00-overview.md`, "Open
questions" #3.

This is a v0.2-or-later feature; doesn't affect v0.1 wire format
since the addition is in the schemaful header (already extension-
gated).

### A.5 Streaming mode

The current spec assumes whole-tensor decode-into-memory. For
training pipelines that stream weights, decoding tensor-by-tensor
without buffering the whole document would be useful. Same wire
format; different consumer API.

## B. JSON profile (weavepack-json)

### B.1 Move/copy ops

JSON Patch (RFC 6902) defines `move` and `copy` operations for
relocating values within a tree. The JSON profile rejects these by
design ("no conformance theater"), but a future revision could add
them under a different op name if the wire-format savings are real.

Citation: `weavepack/profiles/json/04-deltas.md`, "Known limitations"
#1 + ROADMAP.md design philosophy.

The argument against: move/copy are syntactic sugar for
remove + add; they don't save bits. Reject unless someone
demonstrates a measurable win.

### B.2 Splice on object arrays (per-element diff)

Currently arrays of objects fall back to whole-array replace
because partial diffs of nested objects within an array are too
brittle. A more sophisticated differ could emit per-object diffs.

Citation: `weavepack/profiles/json/04-deltas.md`, splice section.

### B.3 Range delete for object keys

Bulk-delete primitives don't exist for objects (object keys have
no positional ordering, so no equivalent to array splice). A
future revision could add a "delete keys matching predicate" op,
but the protocol-side savings are unclear vs N individual remove
ops.

## C. Core protocol (weavepack-core)

### C.1 Schema sidecar protocol

`weavepack/core/06-schemas.md` defines the *concept* but most
details (schema language, hash function, sidecar fetching) are
profile-specific. The tensor profile has implemented schemaful
encoding (5.5 done); JSON has not. Whether JSON benefits from
schemas is an open question — JSON's strmap dedup already covers
the most common case (repeated keys).

### C.2 Schema-as-delta-chain

The mention of recursive meta-schemas (a schema that's itself a
weavepack delta chain). Cited as "the most ambitious version of
the schema feature" in core/06-schemas.md. Deferred to Phase 5+.

Concrete use case: long-lived database tables where the schema
itself evolves over time, but old payloads must remain decodable.
This is what Avro tries to do; weavepack could do it cleaner via
delta chains.

### C.3 Explicit profile-id field

v1.x payloads have an implicit profile-id (= 0, JSON). To use a
non-JSON profile, payloads currently route through the extension
gate. v2 (or v1.2) will add an explicit profile-id in the wire
envelope.

Citation: `weavepack/core/07-extensions.md`.

### C.4 Streaming-mode conformance

The conformance corpus assumes whole-payload encode/decode.
Streaming-mode (incremental decode of a large payload) needs a
separate vector category. Currently not specified.

Citation: `weavepack/core/09-conformance.md`, Open issues #2.

### C.5 Cross-architecture endianness validation

Spec mandates little-endian for multi-byte fields. All current
implementations and test machines are LE. Big-endian validation
needs a CI runner on a BE host (Linux ppc64, s390x).

Citation: `weavepack/profiles/tensor/05-conformance.md`, Open
issues #1.

## D. Implementation gaps (not spec changes; just work to do)

### D.1 Rust JSON encoder

The Rust JSON crate has a Level 1+2 decoder but no encoder. Adding
the encoder gives Rust full L3 conformance for JSON.

Citation: `impl/rust/weavepack-json/` README + Phase 6.2 status in
ROADMAP.md.

### D.2 Python JSON structured-mode

Python decoder currently handles only single-payload mode (36/93
vectors). The remaining 57 are structured-mode containers + deltas.

Citation: `impl/python/README.md` + cross-language conformance
output.

### D.3 Independent third-party implementation

All current impls are derived from the JS reference. A genuine
third-party impl (someone else writing from spec without consulting
JS source) would validate the spec is implementable independently.

The Python PoC partially does this (was written from prose only),
but is by the same maintainer. A truly independent third-party
impl in any language remains an open invitation — see
`weavepack/governance/02-profile-registry.md`.

## E. Governance / operational items

### E.1 Registry maintainer role

Currently TBD per `weavepack/governance/02-profile-registry.md`.
Until appointed, the role rotates informally. When the ecosystem
hits 5+ profiles, a permanent maintainer becomes useful.

### E.2 Badge endpoint

Conformance badges (`[![weavepack-json L3]...](...)`) are
referenced in `weavepack/governance/04-conformance-certification.md`
but the badge URL endpoint doesn't exist yet. To set up: a small
service (e.g. shields.io custom endpoint) that reads the
implementation registry + the most recent conformance run output
and renders the corresponding SVG.

### E.3 Cloud-based CI

Each implementation should run conformance in CI on every commit.
Currently only the JS implementation has CI infrastructure (npm
test in package.json). The Rust crates and Python PoC need GitHub
Actions setups.

## Prioritization (suggested)

If we were to slot v0.2:

**Tier 1 (ship-blocking for v0.2):**
- A.1 fp16/bf16 implementation — RFC 0001 acceptance
- A.2 region_replace — fills the dense-block-update gap
- A.3 delta-from-prior — the headline ML-training feature
- D.1 Rust JSON encoder — completes Phase 6.2 properly
- D.2 Python JSON structured-mode — closes the cross-language
  conformance gap

**Tier 2 (post-v0.2):**
- A.1 int4/uint4, fp8 (less urgent dtypes)
- A.4 sub-tensor random access (consumer optimization)
- B.2 array-of-objects per-element diff (JSON differ improvement)
- C.1 JSON schema profile (only if benchmarks justify)

**Tier 3 (research-grade):**
- A.5 streaming mode
- C.2 schema-as-delta-chain
- C.4 streaming conformance corpus
- C.5 cross-arch endianness validation

**Operational (parallel):**
- E.1 Registry maintainer role appointment (when Tier 1 ships)
- E.2 Badge endpoint (when ≥3 impls claim L3)
- E.3 CI setup (immediate; doesn't require v0.2 work)

## Estimated v0.2 timeline

If v0.1 is the current state and we commit to Tier 1 only:

- A.1 fp16/bf16 wire-up to encode/decode pipelines: 2-3 weeks
- A.2 region_replace impl + tests: 1-2 weeks
- A.3 delta-from-prior arithmetic: 2-3 weeks (including benchmarks)
- D.1 Rust JSON encoder: 3-4 weeks
- D.2 Python JSON structured-mode: 1-2 weeks (translation from
  Rust)
- Test corpus extensions: 1 week
- Documentation updates: 1 week

Total: 11-16 weeks of focused work, OR 4-6 months at one item per
person-month.

The biggest item by absolute time is the Rust JSON encoder; it
requires careful porting of the JS encoder's column accumulator
logic.

## What v0.2 doesn't include

Explicitly out of scope for v0.2 (revisit for v1.0+):

- Formal verification (Lean, Coq) — not justified by adoption
- Streaming mode — not requested by any current consumer
- Standardization (RFC submission to IETF) — premature
- Trademark / branding — community-organic, not formal
- Custom-dtype extension mechanism — reserved-id slots cover this
  for now

## Process

This planning doc gets revised as features ship. When a Tier 1
item lands, mark it in this file (or remove if it's truly done).
When new deferrals are recorded in spec docs, add them here so
this stays the single source of truth for "what's deferred".

Update cadence: refresh after each major milestone (e.g. RFC
acceptance, new phase completion).
