# Weavepack v0.7 — planning document

**Status:** Planning. Defines the scope for the v0.7 cycle.
**Precondition:** v0.6 complete (2026-05-12). All v0.6 benchmark gates pass.

## What v0.7 is

**Profile #7: weavepack-ast** — the primary deliverable. A code/syntax-tree
format with native subtree-delta chains. No existing syntax-tree format has
protocol-level delta semantics; the incumbent (tree-sitter binary + re-parse
on every edit) re-encodes the entire tree for each mutation.

## Why weavepack-ast

The V0.5-PLANNING.md deferred weavepack-ast pending "whether log's
schema-evolution model generalizes to tree-shaped data." That question is
now settled:

1. **Graph (v0.6) confirmed the node/parent-ref model works.** weavepack-graph
   proved that nid-addressed columnar node blocks with typed property columns
   encode graph topology compactly. An AST is a strictly simpler structure
   than a general property graph (each node has exactly one parent), so the
   same column machinery applies with fewer edge cases.

2. **Tree-specific delta ops fill the gap graph does not.**  Graph deltas
   address edges (src/dst). AST deltas address tree positions (parent_nid +
   child_index). Three AST-specific ops — `node_move`, `kind_rename`, and
   `subtree_replace` — have no natural graph counterpart and cover the most
   common code-editing patterns:
   - **node_move**: cut-paste of a function or expression (O(1) op vs tens of
     line-diff hunks)
   - **kind_rename**: rename a symbol globally (one op vs N prop_set calls)
   - **subtree_replace**: atomic replacement of any subtree

3. **AI/ML code tooling is a growing adopter.** LLM code generation, diff-
   application, and refactoring pipelines increasingly work at the AST level
   (Babel transforms, tree-sitter queries, LSP tree manipulation). A compact
   delta format for ASTs is directly useful for streaming tree diffs between
   a language server and an editor, or between a code-generation model and a
   compilation cache.

weavepack-ast's unique value proposition against JSON-encoded AST + gzip
(the de-facto standard: Acorn / Babel / tree-sitter all serialize to JSON):

| | typed | schema | deltas | column-enc | tree-ops | self-desc |
|---|---|---|---|---|---|---|
| Acorn JSON | ✗ | ✗ | ✗ | ✗ | ✗ | ✓ |
| Babel JSON | ✗ | ✗ | ✗ | ✗ | ✗ | ✓ |
| tree-sitter binary | ✓ | ✗ | ✗ | ✗ | ✗ | ✗ |
| ESTree + JSON Patch | ✗ | ✗ | ~ | ✗ | ✗ | ✓ |
| **weavepack-ast** | **✓** | **✓** | **✓** | **✓** | **✓** | **✓** |

The win is sharpest for two scenarios:

1. **Symbol rename** — rename a variable across 50 call sites generates 50
   `prop_set` ops, each ~8 bytes (nid + new name). JSON Patch / AST-diff tools
   emit one patch per affected line, carrying the entire line context. A
   1 000-node AST with 50 renamed nodes: ~400 bytes weavepack delta vs ~15 KB
   AST-diff JSON. The `kind_rename` op collapses this to a single 10-byte
   frame when the new name is short.

2. **Edit stream** — a language server applying 200 incremental edits (function
   additions, deletions, symbol renames, argument reorders) to a file. JSON
   tools re-serialize the full AST on every edit checkpoint. weavepack emits
   only the changed nodes per frame: each `node_insert` or `prop_set` carries
   the columnar delta, O(changed_nodes) bytes per frame.

## Profile model

weavepack-ast is a **labeled property tree** profile:

- **Nodes** carry a `nid` (uint64, monotone), a `parent_nid` (node_id,
  nullable; NULL for root), a `child_index` (uint32, position among siblings),
  a `kind` (string, interned), and named typed property columns.
- Data is organized into typed **node_blocks** grouping nodes by kind.
  All-kind encoding (mixing kinds in one block) is also supported when
  heterogeneous insertion is needed.
- A **tree document** is a sequence of node_blocks, optionally preceded
  by a schema sidecar.

Tree invariants:
- Each nid is unique across the document.
- Every non-root node's `parent_nid` must reference an nid present in
  the document (either in the same payload or in the live state when
  applied as a delta chain).
- `child_index` values for siblings with the same parent must be unique
  and contiguous starting at 0. The encoder assigns them in insertion order;
  `node_move` and `node_insert` preserve this invariant.

Relationship to weavepack-graph:
- weavepack-ast is **not** a subset of weavepack-graph. Both use nid-addressed
  columnar node blocks, but AST has mandatory parent/child-index columns
  (no edge blocks), tree-specific delta ops, and a different wire layout.
- Profile isolation is maintained: `sdk/src/profiles/ast/` imports zero code
  from `sdk/src/profiles/graph/`.

### 01-types.md

Column types (same 16 ctypes as weavepack-graph):

| ctype | Name | Bits | Notes |
|---|---|---|---|
| 0 | `bool` | 1 | |
| 1 | `int8` | 8 | signed |
| 2 | `int16` | 16 | signed |
| 3 | `int32` | 32 | signed |
| 4 | `int64` | 64 | signed |
| 5 | `uint8` | 8 | |
| 6 | `uint16` | 16 | |
| 7 | `uint32` | 32 | |
| 8 | `uint64` | 64 | |
| 9 | `float32` | 32 | IEEE 754 binary32, little-endian |
| 10 | `float64` | 64 | IEEE 754 binary64, little-endian |
| 11 | `string` | variable | LEB128-byte-count + UTF-8 bytes |
| 12 | `bytes` | variable | LEB128-byte-count + raw octets |
| 13 | `date32` | 32 | days since Unix epoch |
| 14 | `timestamp64` | 64 | microseconds since Unix epoch (UTC) |
| 15 | `node_id` | 64 | uint64 with semantic type "node reference" |

`node_id` (ctype 15) signals a parent reference or cross-tree reference.
Same 8-byte LE encoding as `uint64`; distinguished at the type level so
traversal APIs can follow parent_nid chains without schema.

Nullability: companion null bitmap per nullable column (same as graph).

### 02-containers.md

Container model:

| container | notes |
|---|---|
| `node_block` | N nodes with the same `kind`; columnar |
| `mixed_block` | N nodes with mixed kinds; carries a per-row kind column |
| `tree` | a sequence of node_blocks and/or mixed_blocks |

**node_block** mandatory columns:

- col_id 0: `nid` (uint64) — node identifiers, monotone, delta-coded
- col_id 1: `parent_nid` (node_id, nullable) — parent node; NULL for root
- col_id 2: `child_index` (uint32) — position among siblings (0-based)
- col_id 3: `kind` (string) — node kind; interned in strmap

User-defined property columns use col_ids ≥ 4.

Common property col_ids (by convention, not required):
- col_id 4: `name` (string, nullable) — identifier / label name
- col_id 5: `value` (string, nullable) — literal value (string / number)
- col_id 6: `start` (uint32, nullable) — source offset (bytes)
- col_id 7: `end` (uint32, nullable) — source offset end (bytes)

**mixed_block** adds a per-row kind column (col_id 3 is present but is
a `string` column rather than a block-level constant). Useful when a
delta inserts nodes of heterogeneous kinds (e.g., a statement list
containing ExpressionStatement, IfStatement, and ReturnStatement).

Block encoding follows the same wire layout as weavepack-graph node_block:
- 1-byte block type tag (0x00 = node_block, 0x01 = mixed_block)
- LEB128 num_nodes
- LEB128 num_cols
- For each column: LEB128 col_id, 1-byte ctype, nullable flag, data

nid column (col_id 0) uses delta-pack (monotone uint64, same as graph).
parent_nid (col_id 1) is plain uint64 LE (8 bytes each), nullable.
child_index (col_id 2) uses delta-pack (monotone uint32 within a sibling group; packed as LEB128 deltas). The encoder SHOULD sort nodes by (parent_nid, child_index) within a block to maximise delta compression.

### 03-paths.md

Path grammar for delta operations:

- `node/N` — a node by nid (uint64)
- `node/N.prop_name` — a property of a node, by name
- `node/N[col_id]` — a property of a node, by col_id
- `@nid` — the mandatory nid column (col_id 0)
- `@parent` — the mandatory parent_nid column (col_id 1)
- `@child_index` — the mandatory child_index column (col_id 2)
- `@kind` — the mandatory kind column (col_id 3)
- `kind:K` — all nodes with kind string K (schema-optional; scanned)
- `.KindName` — schema-addressed: all nodes of a named kind (schema required)
- `.KindName.prop_name` — schema-addressed: property column on a kind

PATH_KIND constants (wire encoding, 4-bit field):

| Code | Name | Description |
|---|---|---|
| 0 | NODE | Node by nid |
| 1 | NODE_COL | Node by nid + col_id |
| 2 | NODE_KIND | All nodes with a given kind string |
| 3 | AT_NID | Mandatory nid column |
| 4 | AT_PARENT | Mandatory parent_nid column |
| 5 | AT_CHILD_INDEX | Mandatory child_index column |
| 6 | AT_KIND | Mandatory kind column |
| 7 | NODE_PROP | Node by nid + prop name |
| 8–15 | reserved | MUST reject with `unknown_path_kind` |

### 04-deltas.md

Delta operations:

| Op | Code | Description |
|---|---|---|
| `node_insert` | 0 | Insert N AST nodes (columnar; parent_nid + child_index define positions) |
| `node_delete` | 1 | Remove nodes by nid (and implicitly their subtree descendants) |
| `node_move` | 2 | Move a subtree to a new parent_nid and child_index |
| `prop_set` | 3 | Update a property value for a node |
| `kind_rename` | 4 | Rename all nodes whose kind matches a source string to a new kind string |
| `subtree_replace` | 5 | Atomically replace the subtree rooted at a given nid |

Codes 6–7 are reserved. A decoder reading an unknown op code MUST
refuse with `unknown_delta_op`.

Op semantics:

**node_insert**: Columnar insert of N nodes into the live tree state. Each
inserted node's `parent_nid` must already exist in the live state (or be
another node in the same `node_insert` block, provided topological order is
preserved). The encoder MUST topologically sort nodes within a block
(parent before child). `child_index` values must not collide with existing
children of the target parent; the encoder renumbers existing siblings if
necessary and emits the shifted siblings' updated `child_index` values in
a companion `prop_set` frame.

**node_delete**: Takes a list of nids (delta-packed uint64 LEB128). Deletes
each named node and recursively all descendants in the live tree. If a
referenced nid does not exist in the live state, the op is a no-op for
that nid (idempotent delete).

**node_move**: Takes `nid` (uint64), `new_parent_nid` (uint64, 0 = make root),
`new_child_index` (uint32). Moves the subtree rooted at `nid` to the new
position. The encoder must renumber displaced siblings at both source and
destination parents; displaced sibling updates are embedded in the move frame
as a compact delta list rather than emitted as separate `prop_set` frames.

**prop_set**: Same semantics as weavepack-graph prop_set. Takes a path
(PATH_KIND + nid/kind + col_id/name), a ctype, nullable flag, and a new
value. `kind_rename` via prop_set is legal but less efficient than the
dedicated op when renaming many nodes.

**kind_rename**: Takes `old_kind` (LEB128-prefixed UTF-8 string) and
`new_kind` (LEB128-prefixed UTF-8 string). Renames every node in the live
tree whose `kind` equals `old_kind` to `new_kind`. Encoded as a single frame
regardless of how many nodes are affected. The decoder iterates the live
node table and applies the rename in O(N) time. This is the structural
advantage of weavepack-ast over JSON Patch: a rename affecting 1 000 nodes
costs O(1) wire bytes, not O(1 000).

**subtree_replace**: Takes `root_nid` (uint64) and an inline node_block
payload encoding the replacement subtree. The decoder deletes all
descendants of `root_nid` (keeping `root_nid` itself) then inserts the
new subtree nodes as children of `root_nid`. Useful for atomic refactors
(e.g., replace an entire function body).

Delta composition laws:
- `node_insert` ops with disjoint nid sets commute.
- `node_delete` is idempotent; delete-then-delete on the same nid = no-op.
- `node_move` followed by `node_delete` on the same nid = `node_delete`.
- `prop_set` twice on the same (nid, col_id) = last-write-wins.
- `kind_rename` is idempotent for the same (old_kind, new_kind) pair.
- `subtree_replace` twice on the same root_nid = last-write-wins.

### 05-conformance.md

Test corpus structure:

- **Level 1** (block only): encode + decode a node_block. Round-trip.
  Covers all ctypes, nullable parent_nid, kind column.
- **Level 2** (single delta): node_insert or node_delete + one read-back.
  Verifies tree state after one op.
- **Level 3** (delta chain): multi-frame chain, all 6 op types.
  Covers node_move, kind_rename, subtree_replace chains.

### 06-schemas.md

Schema sidecar (parallel to graph 06-schemas.md):

- Declares node kinds with their column schemas (col_ids ≥ 4)
- Schema format: canonical JSON (sorted keys, no trailing decimals)
- Schema hash: SHA-256 of canonical JSON bytes
- Schema-addressed paths (`.KindName.prop_name`) require the schema

Schema example (ESTree-style JS AST subset):

```json
{
  "kinds": {
    "Program": { "cols": [{"id":4,"name":"sourceType","ctype":11,"nullable":false}] },
    "FunctionDeclaration": { "cols": [{"id":4,"name":"name","ctype":11,"nullable":false}] },
    "Identifier": { "cols": [{"id":4,"name":"name","ctype":11,"nullable":false}] },
    "Literal": {
      "cols": [
        {"id":4,"name":"value","ctype":11,"nullable":true},
        {"id":5,"name":"raw","ctype":11,"nullable":false}
      ]
    }
  }
}
```

### 07-benchmarks.md

Benchmark methodology and results vs. Acorn/ESTree JSON + gzip.

Target scenarios:

1. **Snapshot**: encode a 1 000-node synthetic JS AST (FunctionDeclarations,
   BlockStatements, ExpressionStatements, Identifiers, Literals). Node schema:
   (nid uint64, parent_nid node_id, child_index uint32, kind string, name
   string nullable, value string nullable, start uint32, end uint32).
   Compare weavepack-ast snapshot vs. ESTree JSON + gzip-6.
   Gate: weavepack brotli-6 within 3× of ESTree JSON + gzip-6
   (JSON gzip has strong back-reference advantage on `"type"`, `"name"`,
   `"start"`, `"end"` key repetition).

2. **Symbol rename (50 renames)**: Start from the snapshot, emit 50
   `kind_rename` or `prop_set` delta frames renaming one identifier name
   across 50 occurrences. Compare cumulative stream vs. per-frame
   ESTree JSON + gzip-6 re-snapshot.
   Gate: weavepack raw ≥ 50× smaller than per-snapshot ESTree JSON + gzip.

3. **Edit stream (200 edits)**: 200 mixed-op delta frames: node_insert (5
   nodes), node_delete (2 nodes), prop_set (update `name`), or node_move
   (1 subtree). Compare cumulative stream vs. per-frame ESTree JSON + gzip-6.
   Gate: weavepack raw ≥ 10× smaller than per-snapshot ESTree JSON + gzip.
