# Weavepack v0.4 — planning document

**Status:** Planning. Defines the scope for the v0.4 cycle.
**Precondition:** v0.3 complete (2026-05-08). All v0.3 wire-profile items shipped.

## What v0.4 is

Two parallel threads:

1. **Profile #4: weavepack-tabular** — the primary deliverable. A
   Parquet/Arrow-equivalent column-store format with native row-level
   delta chains. No existing tabular format has per-update row deltas;
   this is the gap.

2. **Carry-forward items from v0.3** — see §Carry-forward below. Treated
   as secondary; they improve coverage without distracting from the main
   goal.

## Why weavepack-tabular

The ROADMAP ranks profiles by strategic value:

1. weavepack-tensor ✓ shipped
2. weavepack-wire ✓ shipped
3. **weavepack-tabular** — analytics / CDC market; unique delta story

weavepack-tabular's unique value proposition against Parquet + brotli:

| | bit-pack | deltas | schema | self-desc | streaming |
|---|---|---|---|---|---|
| Parquet | ✓ (RLE/Dict) | ✗ | ✓ | ✗ | ✗ |
| Arrow IPC | ✗ (byte-aligned) | ✗ (append-only) | ✓ | ✗ | ✓ |
| ORC | ✓ | ✗ | ✓ | ✗ | ✗ |
| **weavepack-tabular** | **✓** | **✓** | **✓** | **✓** | **✓** |

The win is sharpest for **Change-Data-Capture (CDC) streams** — a
database replication log where each event touches 1–5 columns of a
wide row. Existing CDC consumers must re-serialize the entire row per
event (Debezium, Kafka Connect). weavepack-tabular emits a `row_update`
delta that encodes only the changed columns.

A secondary win: **streaming analytics** (append-only ingestion with
periodic aggregation). Arrow IPC supports append but not row-level
updates or schema evolution. weavepack-tabular adds `column_add`,
`column_drop`, and `batch_upsert` at the wire level.

Estimated compression win over Parquet+brotli for CDC streams:
same structural delta advantage as tensor/wire (3–18×) when only a
small fraction of columns change per update.

## Profile #4 spec plan

Parallel structure to the wire profile. All documents live under
`weavepack/profiles/tabular/`.

### 00-overview.md

Context, motivation, incumbent comparison, open design questions.

### 01-types.md

Column types (analogous to Parquet/Arrow scalar types):

| ctype | Name | Bits | Notes |
|---|---|---|---|
| 0 | `bool` | 1 | |
| 1 | `int8` | 8 | signed two's-complement |
| 2 | `int16` | 16 | signed |
| 3 | `int32` | 32 | signed |
| 4 | `int64` | 64 | signed |
| 5 | `uint8` | 8 | |
| 6 | `uint16` | 16 | |
| 7 | `uint32` | 32 | |
| 8 | `uint64` | 64 | |
| 9 | `float32` | 32 | IEEE 754 binary32, little-endian |
| 10 | `float64` | 64 | IEEE 754 binary64, little-endian |
| 11 | `string` | variable | LEB128-byte-count + UTF-8 bytes |
| 12 | `bytes` | variable | LEB128-byte-count + raw octets |
| 13 | `date32` | 32 | days since Unix epoch (1970-01-01) |
| 14 | `timestamp64` | 64 | microseconds since Unix epoch (UTC) |
| 15 | `null` | 0 | all-null column sentinel (schema-declared) |

Nullability: each non-null column carries a companion **null bitmap**
(1 bit per row); if a row's bit is 1 the cell is NULL and the value
column carries no bits for that row.

Row identity: each frame assigns a monotonically-increasing `uint64`
row_id beginning at 0. Delta ops address rows by row_id.

### 02-containers.md

Container model:

| container | notes |
|---|---|
| `frame` | the top-level document: a set of typed columns + row_ids |
| `column` | a typed, named, nullable column within a frame |
| `row` | a logical cross-column record identified by row_id |

A frame is encoded as:
- `schema_hash`: 32 bytes SHA-256 (all-zero = no schema)
- `num_rows`: LEB128 uint64
- `num_cols`: LEB128 uint32
- Per-column: col_id (uint32), ctype (4 bits), nullable (1 bit),
  null_bitmap (if nullable: `num_rows` bits), value column

Value column encoding follows the weavepack-core typed column layout
(bit-packed values per ctype, with RLE on runs of identical values).

### 03-paths.md

Path grammar for delta operations:

- `[col_id]` — column by integer id (always works)
- `.col_name` — column by name (schema required)
- `#row_id` — a specific row (used in update/delete ops)
- `#row_id[col_id]` — a specific cell

### 04-deltas.md

Delta operations (parallel to wire 04-deltas.md):

| Op | Code | Description |
|---|---|---|
| `row_insert` | 0 | Insert N new rows with values for all columns |
| `row_update` | 1 | Update specific column values for given row_ids |
| `row_delete` | 2 | Remove rows by row_id |
| `column_add` | 3 | Add a new column (existing rows get NULL or default) |
| `column_drop` | 4 | Remove a column by col_id |
| `column_rename` | 5 | Rename a column (schema-layer change only) |
| `batch_upsert` | 6 | Insert-or-update by row_id (CDC use case) |

Delta composition laws:

- `row_insert` then `row_delete` on same row_id = empty (net-zero)
- `row_update` twice on same cell = last-write-wins
- `column_add` then `column_drop` same col_id = net-zero
- `batch_upsert` is idempotent: multiple upserts on same row_id converge

### 05-conformance.md

Test corpus structure:

- **Level 1** (snapshot only): encode + decode a frame. Round-trip.
- **Level 2** (single delta): snapshot + one op, verify final state.
- **Level 3** (delta chain): multi-frame chain, all 7 op types.

### 06-schemas.md

Schema sidecar (parallel to tensor/wire 06-schemas.md):

- Declares column ids, names, types, nullability
- Schema format: canonical JSON (sorted keys, no trailing decimals)
- Schema hash: SHA-256 of canonical JSON bytes
- Without schema: column names unavailable; type inferred from ctype
- With schema: name-addressed paths, type validation at decode

### 07-benchmarks.md

Benchmark methodology and results vs. Parquet + brotli.

Target scenarios:

1. **Full snapshot**: 10-column table, 10 000 rows. Compare snapshot
   size vs. Parquet (snappy) + brotli.

2. **CDC stream**: 20-column table, 5 000 rows, 1 000 update events,
   each touching 1–3 columns. Compare chain size vs. (Parquet row-group
   × 1 000) + brotli.

3. **Append stream**: 1 000 `row_insert` events, 10 rows each. Compare
   chain size vs. (Parquet row-group × 1 000) + brotli.

Gate: weavepack-tabular beats Parquet+brotli by ≥2× on scenario 2 and
scenario 3. Scenario 1 within 20% (Parquet's dict-encoding of repeated
string values gives it an advantage on some column distributions).

## Implementation plan

### Step T.1 — Spec docs 00–04 (this cycle)

Write spec docs 00-overview.md through 04-deltas.md in
`weavepack/profiles/tabular/`. No implementation yet; spec first.

### Step T.2 — JS reference implementation

`sdk/src/profiles/tabular/` — parallel to `sdk/src/profiles/wire/`.

- `encoder.js` — frame + delta encoder
- `decoder.js` — frame + delta decoder
- `index.js` — exports `encodeFrame`, `decodeFrame`, `encodeChain`, `decodeChain`

All existing tests must continue to pass (profile isolation).

`weavepack/profiles/tabular/test-vectors/`:

- `types/scalars.json` — round-trips for all ctypes
- `types/nulls.json` — nullable columns, null bitmaps
- `containers/frames.json` — multi-column frames
- `containers/schema.json` — schema-addressed paths
- `deltas/row_ops.json` — row_insert, row_update, row_delete
- `deltas/col_ops.json` — column_add, column_drop, column_rename
- `deltas/upsert.json` — batch_upsert chain

Extend `weavepack/tools/verify-test-vectors.js` with a tabular profile
handler.

## What v0.4 does NOT include

- **weavepack-log**: deferred to v0.5. Log profile benefits from the
  row-delta machinery delivered in tabular; design it after tabular
  proves the row-delta model in practice.
- **weavepack-graph**: deferred. RDF/property-graph delta semantics
  require deeper research; no strong adopter demand yet.
- **IETF standardization**: premature; requires ≥2 independent
  certified impls per profile first.

## Benchmark gate (v0.4 ship criterion)

v0.4 ships when all of these hold:

1. weavepack-tabular spec docs 00–07 are written and consistent.
2. JS + Rust + Python implementations pass all conformance vectors.
3. Benchmark scenario 2 (CDC stream): chain is ≥2× smaller than
   Parquet+brotli for 1 000 update events touching 1–3 of 20 columns.
4. Benchmark scenario 3 (append stream): same ≥2× criterion.
5. Benchmark scenario 1 (full snapshot): within 20% of Parquet+brotli.
