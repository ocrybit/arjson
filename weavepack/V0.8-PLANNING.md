# Weavepack v0.8 — planning document

**Status:** Planning. Defines the scope for the v0.8 cycle.
**Precondition:** v0.7 complete (2026-05-12). All v0.7 benchmark gates pass.

## What v0.8 is

**Profile #8: weavepack-geo** — the primary deliverable. A GeoJSON-compatible
binary format for geographic features with native delta chains and typed
columnar property encoding.

## Why weavepack-geo

The V0.7-PLANNING.md listed geo as the next planned profile after ast. The
motivating factors:

1. **GeoJSON is universally used but notoriously bloated.** RFC 7946 GeoJSON
   is human-readable JSON: coordinate pairs repeat their floating-point text
   encoding for every vertex (e.g. `[37.7749,-122.4194]`). A city boundary
   polygon with 50 000 vertices occupies ~2 MB of GeoJSON text. The same
   vertices as raw float64 binary require ~800 KB (2.5×). With columnar
   storage and brotli compression on homogeneous float64 columns, 150–200 KB
   is achievable (10–15×).

2. **No existing binary geo format has protocol-level delta chains.** The
   incumbents:

   | Format | Binary | Schema | Deltas | Column-enc | Self-desc |
   |---|---|---|---|---|---|
   | GeoJSON | ✗ | ✗ | ✗ | ✗ | ✓ |
   | WKB | ✓ | ✗ | ✗ | ✗ | ✗ |
   | GeoPackage | ✓ | ✓ | ✗ | ✗ | ~ |
   | Flatbuffers/GeoArrow | ✓ | ✓ | ✗ | ✓ | ✗ |
   | MVT/Mapbox | ✓ | ~ | ✗ | ✗ | ✗ |
   | **weavepack-geo** | **✓** | **✓** | **✓** | **✓** | **✓** |

   Live map tiles and streaming geographic updates (ride-sharing positions,
   weather radar, seismic sensor streams) currently transmit full-polygon or
   full-feature-collection snapshots on every update. weavepack-geo’s
   `feature_insert` / `geometry_replace` / `prop_set` delta chain transmits
   only changed features per frame.

3. **Profile isolation is cleanly achievable.** The geo profile needs only
   the core ctype table (float64 for coordinates, string for feature ids and
   properties, uint32/uint64 for ring/part counts). It does not borrow from
   json, graph, or ast profile code.

## Profile model

weavepack-geo encodes **GeoJSON-compatible feature collections** as a sequence
of typed **feature blocks**:

- A `feature_block` groups N features that share the same **geometry type**
  (POINT, LINESTRING, POLYGON, MULTIPOINT, MULTILINESTRING, MULTIPOLYGON) and
  the same set of **property column names and types**.
- Within a feature_block, coordinates are stored in separate typed columns:
  `x` (float64), `y` (float64), and optionally `z` (float64). This gives
  compressors a homogeneous float64 stream per dimension instead of the
  interleaved GeoJSON pairs.
- Property columns are typed (same 16 ctypes as tabular/graph/ast) and
  nullable via MSB-first null bitmaps.
- A `geometry_collection_block` encodes GEOMETRY_COLLECTION features where
  each feature holds a heterogeneous sub-geometry list.

A **weavepack-geo document** is:

```
document:
  LEB128 num_blocks
  block*
```

where each `block` is one of: `feature_block`, `geometry_collection_block`,
`delta_frame`.

## Geometry encoding strategy

GeoJSON has three layers of coordinate nesting for polygons and multi-
geometries. weavepack-geo flattens these into three parallel arrays:

| Layer | GeoJSON | weavepack-geo encoding |
|---|---|---|
| Coordinates | `[x, y]` pairs | Separate `x` and `y` float64 columns (+ optional `z`) |
| Ring/part counts | implicit (nested JSON arrays) | Per-feature `ring_counts` and `coord_counts` uint32 arrays |
| Multi-part counts | implicit | Per-feature `part_count` uint32 value |

This flatting removes all JSON nesting overhead while preserving the full
GeoJSON topology.

For POLYGON and MULTIPOLYGON, each ring (including exterior ring and interior
“holes”) stores its vertex count in the `ring_counts` array. The first ring
is always the exterior ring; subsequent rings are interior rings (holes). The
sum of all ring_counts for a feature equals the total vertex count for that
feature.

For MULTILINESTRING and MULTIPOLYGON, a `part_count` per feature gives the
number of parts (line parts or polygon parts). For MULTIPOLYGON, the
ring_counts span all rings across all parts; the decoder uses part_count +
per-part ring counts (an additional `part_ring_counts` sub-array) to
reconstruct the nesting.

## Coordinate precision

weavepack-geo supports two coordinate precisions:

- **full** (default): IEEE 754 float64, 8 bytes per coordinate, ~1 nanodegree
  precision (~0.11 mm at the equator).
- **reduced**: IEEE 754 float32, 4 bytes per coordinate, ~6 decimal digit
  precision (~1.1 m at the equator). Declared in the block header.

Reduced precision is a block-level flag; it is appropriate for consumer-grade
GPS tracks, map tiles, and visualisation layers where sub-metre accuracy is
not required.

## Property schema

Each feature_block declares a schema: an ordered list of column descriptors
`(name, ctype, nullable)`. Property values are stored in typed columns in
this order. A block with zero property columns is valid (geometry-only features).

Feature IDs (GeoJSON `id` field) are stored as a separate optional column:

- Absent (no `id` declared): the block header carries a `fid_kind = 0` byte.
- String IDs: `fid_kind = 1`; stored as a `string` column.
- Integer IDs: `fid_kind = 2`; stored as a `uint64` column.

## Delta chains

A `delta_frame` contains one or more operations targeting a named collection.
Collections are identified by a LEB128-length + UTF-8 name string.

Operations:

| Op | Code | Description |
|---|---|---|
| `feature_insert` | 0 | Insert N new features into a collection |
| `feature_delete` | 1 | Remove features by fid or by index range |
| `geometry_replace` | 2 | Replace the geometry of a specific feature |
| `prop_set` | 3 | Set or update one property value on a feature |
| `prop_delete` | 4 | Remove a property from a feature |
| `collection_replace` | 5 | Atomic full-snapshot replacement of a collection |

### Step GE.6 — Benchmarks

`weavepack/tools/benchmark-geo.js` and
`weavepack/profiles/geo/07-benchmarks.md`.

Benchmark scenarios:

1. **Single FeatureCollection snapshot**: a 1 000-feature city-boundary
   dataset, each feature a POLYGON with ~200 vertices. Compare weavepack-geo
   (float64 + brotli-6) vs. GeoJSON + gzip-6 and vs. raw WKB.

2. **Streaming GPS track**: a LINESTRING with 10 000 vertices extended 50
   vertices at a time (100 delta frames). Compare delta chain vs. GeoJSON
   snapshot × 100.

3. **Feature property updates**: 500 features, 100 delta frames each changing
   5% of feature properties. Compare chain size vs. GeoJSON snapshot × 100.

## Benchmark gate (v0.8 ship criterion)

v0.8 ships when all of these hold:

1. weavepack-geo spec docs 00–07 are written and internally consistent.
2. JS + Rust + Python implementations pass all conformance vectors.
3. Benchmark scenario 1 (snapshot): weavepack-geo brotli-6 is ≤ 40% of GeoJSON
   gzip-6 size (≥ 2.5× compression advantage on vertex data).
4. Benchmark scenario 2 (GPS track delta): delta chain ≤ 10% of GeoJSON
   snapshot × N for N = 100 append frames (≥ 10× advantage).
5. Benchmark scenario 3 (property updates): delta chain ≤ 5% of GeoJSON
   snapshot × N for N = 100 frames (≥ 20× advantage).
