# Weavepack v0.5 — planning document

**Status:** Planning. Defines the scope for the v0.5 cycle.
**Precondition:** v0.4 complete (2026-05-09). All v0.4 benchmark gates pass.

## What v0.5 is

Two parallel threads:

1. **Profile #5: weavepack-log** — the primary deliverable. A structured
   event-stream format with native delta chains and cursor semantics. No
   existing log format has per-stream column encoding or protocol-level cursor
   checkpoints; this is the gap.

2. **Carry-forward items from v0.4** — see §Carry-forward below. Treated
   as secondary; they improve coverage without distracting from the main goal.

## Why weavepack-log

The weavepack-tabular profile proved the row-delta machinery in an analytics
context (v0.4). weavepack-log applies that same machinery to structured event
streams — a larger and more latency-sensitive market.

The ROADMAP deferred weavepack-log to v0.5 explicitly because it "benefits
from the row-delta machinery delivered in tabular; design it after tabular
proves the row-delta model in practice." That proof is now in hand.

weavepack-log's unique value proposition against JSON Lines + gzip and
OpenTelemetry OTLP:

| | typed | schema | deltas | column-enc | cursor | self-desc |
|---|---|---|---|---|---|---|
| JSON Lines | ✗ | ✗ | ✗ | ✗ | ✗ | ✓ |
| Logfmt | ✗ | ✗ | ✗ | ✗ | ✗ | ✓ |
| NDJSON | ✗ | ✗ | ✗ | ✗ | ✗ | ✓ |
| OpenTelemetry OTLP | ✓ | ✓ | ✗ | ✗ | ✗ | ~ |
| **weavepack-log** | **✓** | **✓** | **✓** | **✓** | **✓** | **✓** |

The win is sharpest for two scenarios:

1. **High-repetition log streams** — application logs where 60–80% of fields
   are identical across consecutive events (same host, service, env, level).
   JSON Lines encodes every field name and value in every event. weavepack-log
   encodes field names once per schema and bit-packs values; repeated fields
   compress to near-zero per event via column RLE.

2. **Streaming append pipelines** — consumers that tail a log stream and
   process events in micro-batches. Without delta semantics, each consumer
   checkpoint requires re-encoding all events since the last full snapshot.
   weavepack-log emits `event_append` delta frames containing only the new
   events, giving O(new_events) bytes per micro-batch vs O(all_events)
   for a re-snapshot.

The cursor-checkpoint primitive (protocol-level, not application-level) is
a secondary win: it eliminates the application-layer offset-tracking that
Kafka, Kinesis, and similar systems bolt on externally.

Estimated compression win over JSON Lines + gzip for high-repetition streams:
same structural advantage as tabular (3–20×) when 60–80% of field values
repeat across consecutive events within a batch.

## Profile #5 spec plan

### 01-types.md

Column types (superset of tabular ctypes):

| ctype | Name | Bits | Notes |
|---|---|---|---|
| 0 | `bool` | 1 | (inherited from tabular) |
| 1 | `int8` | 8 | signed |
| 2 | `int16` | 16 | signed |
| 3 | `int32` | 32 | signed |
| 4 | `int64` | 64 | signed |
| 5 | `uint8` | 8 | |
| 6 | `uint16` | 16 | |
| 7 | `uint32` | 32 | |
| 8 | `uint64` | 64 | |
| 9 | `float32` | 32 | IEEE 754 binary32, little-endian |
| 10 | `float64` | 64 | IEEE 754 binary64, little-endian |
| 11 | `string` | variable | LEB128-byte-count + UTF-8 bytes |
| 12 | `bytes` | variable | LEB128-byte-count + raw octets |
| 13 | `date32` | 32 | days since Unix epoch (1970-01-01) |
| 14 | `timestamp64` | 64 | microseconds since Unix epoch (UTC) |
| 15 | `null` | 0 | all-null column sentinel |
| 16 | `level` | 3 | severity level; 3-bit packed (see below) |

`level` encoding:
- TRACE = 0, DEBUG = 1, INFO = 2, WARN = 3, ERROR = 4, FATAL = 5
- Values 6–7 are reserved for future extension
- Bit-packed at 3 bits per value (same packing rule as bool columns in core)
- RLE applies across runs of identical level values (high gain: server logs
  are often long runs of INFO-level events)

Nullability: same as tabular — a companion null bitmap for each nullable
column.

Sequence identity: each event in a stream carries a monotonically-increasing
`seq` (uint64, assigned by the encoder). Delta ops address events by seq.
Within a batch, seq values are contiguous: seq_start to seq_start + N - 1.

### 02-containers.md

Container model:

| container | notes |
|---|---|
| `event_batch` | a columnar block of N events with the same schema |
| `event` | a single log event (one row in an event_batch) |
| `stream_header` | declares stream_id, source identifier, schema_hash |

An event_batch is encoded as a tabular-compatible frame with two reserved
mandatory columns:

- col_id 0: `seq` (uint64) — event sequence numbers (monotone, delta-coded)
- col_id 1: `ts` (timestamp64) — event timestamps (monotone, delta-coded)

User-defined fields use col_ids ≥ 2. The schema_hash (32 bytes SHA-256)
covers the full column schema including mandatory cols 0 and 1.

A stream_header is encoded as:
- `stream_id`: 16-byte UUID (raw bytes)
- `source`: LEB128-prefixed UTF-8 string
- `schema_hash`: 32 bytes SHA-256 (all-zero = schemaless)
- `seq_start`: LEB128 uint64 (first seq number in this stream segment)

The stream_header is optional. Streams without a header are anonymous
(no stream_id, no source attribution). Streams with a header support
cursor-checkpoint ops and schema evolution.

Value column encoding follows the weavepack-core typed column layout
(bit-packed per ctype, RLE on runs of identical values). The mandatory
`seq` and `ts` columns use delta-pack (monotone uint64 sequences).

### 03-paths.md

Path grammar for delta operations:

- `[col_id]` — column by integer id (always works)
- `.col_name` — column by name (schema required)
- `#seq` — a specific event by sequence number
- `#seq[col_id]` — a specific cell (event × field)
- `@seq` — special alias for the mandatory seq column (col_id 0)
- `@ts` — special alias for the mandatory timestamp column (col_id 1)
- `seq:[lo,hi]` — inclusive seq range (used in expire ops)

### 04-deltas.md

Delta operations:

| Op | Code | Description |
|---|---|---|
| `event_append` | 0 | Append N events to the stream (primary append op) |
| `field_update` | 1 | Update field values for a specific event by seq |
| `event_expire` | 2 | Mark a seq range as expired (retention / tombstone) |
| `schema_evolve` | 3 | Add, drop, or rename a field for future events |
| `cursor_checkpoint` | 4 | Record a named cursor at a seq position |

Detailed op semantics:

**event_append**: Like tabular `row_insert`, but seq and ts columns are
mandatory. The encoder validates that seq values are strictly greater than
all previously appended seqs. Each event_append frame encodes only the N new
events; a reader accumulates frames to reconstruct the full stream.

**field_update**: Like tabular `row_update` addressed by seq. Used for
corrections (e.g., redacting a field value in a prior event). Addressed by
seq number (not by position). For compliance use cases: field_update on
a `null`-type column erases the field value in the logical view without
destroying the physical chain.

**event_expire**: Marks a range of events as logically expired. Expired
events are retained in the physical chain but excluded from logical views.
Useful for implementing retention policies (e.g., "keep 30 days of events")
without invalidating the chain's hash ancestry. `event_expire` is idempotent:
expiring an already-expired range is a no-op.

**schema_evolve**: Modifies the schema for events appended after this op.
Sub-ops: `column_add`, `column_drop`, `column_rename` (same semantics as
tabular). Events before the schema_evolve retain their original schema;
events after use the evolved schema. The schema_hash in the stream_header
is updated to reflect the new schema.

**cursor_checkpoint**: Records a named cursor at a seq position. A cursor is
a (name, seq) pair that a consumer can use to resume stream processing.
Example: a consumer named "analytics-pipeline" records `cursor_checkpoint(
"analytics-pipeline", seq=50000)` to mark its last processed event.
Idempotent: checkpointing the same name at the same seq is a no-op.

Delta composition laws:

- `event_append` ops are ordered by seq: two appends with non-overlapping
  seq ranges commute; overlapping seqs are illegal.
- `field_update` twice on the same (seq, col_id) = last-write-wins.
- `event_expire` is idempotent and monotone (cannot un-expire events).
- `schema_evolve` ops do not commute (order matters; apply in chain order).
- `cursor_checkpoint` is idempotent at the same (name, seq) pair.

### 05-conformance.md

Test corpus structure:

- **Level 1** (batch only): encode + decode an event_batch. Round-trip.
  Covers all ctypes including `level` (ctype 16).
- **Level 2** (single delta): event_append + one other op; verify state.
- **Level 3** (delta chain): multi-frame chain, all 5 op types.

### 06-schemas.md

Schema sidecar (parallel to tabular 06-schemas.md):

- Declares field ids (col_ids), names, types, nullability
- Mandatory columns (seq, ts) are always included even in schemaless mode
- Schema format: canonical JSON (sorted keys, no trailing decimals)
- Schema hash: SHA-256 of canonical JSON bytes
- Schema multiplexing: a single stream_header may carry a `schema_set`
  (array of schemas keyed by schema_id); events carry a `schema_id` field
  (uint8, col_id 2 when schema multiplexing is enabled) that selects which
  schema applies to that event. This supports mixed-type streams (e.g.,
  HTTP request events + DB query events in the same stream).

### 07-benchmarks.md

Benchmark methodology and results vs. JSON Lines + gzip.

Target scenarios:

1. **High-repetition batch**: 10 000 events from a single service.
   70% of fields are identical across events (host, service, env, level=INFO).
   Varying fields: ts, seq, request_id (string), duration_ms (float32),
   status_code (uint16). Compare event_batch size vs. JSON Lines + gzip.

2. **Streaming append (per-append delta)**: 1 000 event_append operations,
   each appending 10 events. Stream grows from 0 to 10 000 events.
   Compare: weavepack stream (init + 1 000 delta frames) vs.
   JSON Lines (full batch per step, re-encoding all events).
   Gate: weavepack ≥2× smaller than per-batch JSON Lines + gzip.

3. **Multi-schema stream**: 1 000 events mixing 3 event types (HTTP=40%,
   DB=35%, cache=25%). Schema multiplexing. Compare stream size vs.
   JSON Lines (all events as full objects, redundant field names per event).
   Gate: weavepack ≥2× smaller than per-batch JSON Lines + gzip.

Gate: weavepack-log beats JSON Lines + gzip by ≥2× on scenarios 2 and 3
(where delta chains provide structural advantage). Scenario 1 within 20%
of JSON Lines + gzip (single-batch comparison; gzip has strong back-reference
advantage on repetitive text; column encoding + RLE gives weavepack a
different structural advantage that may not dominate at gzip quality 6).

## What v0.5 does NOT include

- **weavepack-graph**: deferred to v0.6+. RDF/property-graph delta semantics
  require deeper research; no strong adopter demand yet.
- **weavepack-ast**: deferred. Code/syntax-tree delta design (rename-as-delta)
  benefits from observing whether log's schema-evolution model generalizes to
  tree-shaped data.
- **IETF standardization**: premature; requires ≥2 independent certified
  implementations per profile first.
- **Network transport layer**: weavepack defines bytes, not how they're sent.
- **Encryption**: payloads can be encrypted before/after weavepack encoding.

## Benchmark gate (v0.5 ship criterion)

v0.5 ships when all of these hold:

1. weavepack-log spec docs 00–07 are written and consistent.
2. JS + Rust + Python implementations pass all conformance vectors.
3. Benchmark scenario 2 (streaming append): chain is ≥2× smaller than
   per-batch JSON Lines + gzip for 1 000 append steps of 10 events each.
4. Benchmark scenario 3 (multi-schema stream): chain is ≥2× smaller than
   per-batch JSON Lines + gzip for 1 000 events across 3 schemas.
5. Benchmark scenario 1 (single batch): within 20% of JSON Lines + gzip
   for 10 000 events.
