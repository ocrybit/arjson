consolidates the most-used links from the spec + governance directories
so you can find what you need without spelunking.

## Build and test

### JS reference

```bash
cd sdk
npm install
npm test                 # 2181 sdk tests
npm run properties       # ~1700 property-based cases
```

### CLI utility

```bash
echo '{"a":1,"b":[1,2,3]}' | node sdk/bin/wpkt-json.js size
# JSON:     19 bytes
# weavepack: 12 bytes
# ratio:    1.58× smaller

echo '{"a":1}' | node sdk/bin/wpkt-json.js encode > /tmp/x.wpkt
node sdk/bin/wpkt-json.js decode < /tmp/x.wpkt
# {"a":1}
```

After `npm install -g arjson`, `wpkt-json` is on PATH.

### Rust crates

```bash
cd impl/rust
cargo test               # all crates
cargo run -p weavepack-tensor --bin conformance   # 31/31 tensor vectors
cargo run -p weavepack-json   --bin conformance   # 93/93 JSON vectors
```

### Python PoC

```bash
python3 impl/python/conformance.py   # 36/36 single-payload vectors
```

### Cross-language conformance

```bash
node weavepack/tools/verify-test-vectors.js   # 132 JSON+tensor vectors
```

## Common contributions

### Fixing a typo or clarifying prose

Just open a PR. No RFC needed for editorial changes that don't affect
behavior. Add a brief commit message explaining what's clearer.

### Reporting a conformance failure

Open an issue with:
- Implementation name + version
- Vector that fails (path + name)
- Expected output
- Actual output
- Steps to reproduce

If the failure is in the JS reference and the spec text is clear,
that's a JS reference bug; we'll fix it. If the JS reference matches
the spec but your implementation doesn't, that's a bug in your
implementation. If the spec text is ambiguous, see
[`weavepack/governance/06-spec-interpretation.md`](./weavepack/governance/06-spec-interpretation.md).

### Adding a test vector

Test vectors live in `weavepack/profiles/<name>/test-vectors/` and are
generated by `weavepack/tools/generate-*-vectors.js`. To add a new
vector:

1. Add the input case to the relevant generator script
2. Re-run the generator (`node weavepack/tools/gen-<...>-vectors.js`)
3. Run the verifier to confirm round-trip
   (`node weavepack/tools/verify-test-vectors.js`)
4. Commit the regenerated vector file + script change together

### Implementing a new feature spec'd in an RFC

See `weavepack/rfcs/` for filed RFCs. The "Reference implementation"
section of each RFC describes what's expected. The dual-implementation
rule (`weavepack/governance/01-rfc-process.md`) requires the feature
in at least 2 languages before the RFC can be Accepted.

### Implementing weavepack from scratch in a new language

1. Read `weavepack/core/` (protocol-level spec, 10 docs)
2. Read the profile spec(s) you want to support
3. Implement bit primitives (`weavepack/core/03-bit-encoding.md`)
4. Implement the wire envelope (`weavepack/core/02-wire-format.md`)
5. Implement the profile-specific encoders/decoders
6. Run conformance: walk
   `weavepack/profiles/<name>/test-vectors/` and validate against
   `expected_bytes_hex` / `expected_decoded`
7. List your implementation in
   `weavepack/governance/05-implementation-registry.md`

The Python PoC at `impl/python/` is intentionally minimal (~120 lines
for the single-payload decoder). It was written from the spec docs
alone, no JS source consulted. Use it as a sanity check that the spec
is implementable from prose.

## Coding conventions

- **JS**: 2-space indent, no semicolons-required. The existing files
  use semicolons; match what's around you.
- **Rust**: standard rustfmt. `cargo fmt` before committing.
- **Python**: PEP 8, 4-space indent, type hints encouraged.
- **Markdown**: 80-column wrap when practical, but don't break code
  blocks or tables.

## Commit conventions

- One logical change per commit
- Subject line ≤ 72 chars, lowercase, imperative mood
- Body explains the *why* (the *what* is in the diff)
- Co-authorship trailer if AI-assisted:
  `Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>`

## What we won't merge

The "no conformance theater" principle from
[`weavepack/ROADMAP.md`](./weavepack/ROADMAP.md) means we won't merge:

- Compatibility shims with other formats (RFC 6902, JSON Patch, etc.)
- Feature additions purely to match what protobuf/CBOR/parquet does
- Changes that break round-trip correctness for any value
- Changes that bypass tests with `--no-verify` or skip-hook flags
- Changes that add features not called for in the roadmap or an
  accepted RFC
- Bot-driven mass refactors that change behavior

We will merge:

- Bug fixes (with regression test)
- Spec clarifications (no behavior change)
- New profile registrations following the procedure
- New language implementations passing conformance
- RFC-driven feature additions after acceptance + dual-impl
- Performance improvements that don't change wire format

## Communication

- **Spec questions / disputes**: open a GitHub issue on the spec repo
- **Implementation questions**: open a GitHub issue on the implementation
  in question
- **Coordination across implementations**: spec repo issues; ping
  maintainers of relevant impls

There is no Slack, Discord, or mailing list yet. The ecosystem is
small enough that GitHub issues suffice. If that changes, we'll set
up something heavier.

## Code of conduct

Be respectful. Disagree about technical decisions; don't disagree
about whether someone deserves to be heard. If you have to ask
whether something is OK, it's probably not.

There is no formal CoC document yet. If a real conflict arises, we'll
adopt one.
